## 1. Executive Summary

Novatio EU FMD Alert Manager allow Marketing Authorization Holders (MAHs) to process, analyze, and resolve safety alerts generated by the European Medicines Verification Organisation (EMVO) & National Medicine Verification System (NMVS). This systems serves as the interface to handle the high volume of daily alerts (sometimes 100s-1000s) to identify root causes, differentiate false alerts from real counterfeits, and fulfill compliance obligations

The platform is built as a Progressive Web App (PWA) with offline-first architecture, displaying alert stored/updated in supabase backend database in realtime, display the root causae information updated in databases by AI agents, provide option to users to mark the alert as resolved, summary dashboards with options to filter based on alert type and status.

Primary objectives:

- Enable real time data display from backend databases (more like a BI tool)
- Provide option for user to update each alert displayed, its status, notes etc.,
- Support Manufacturer → Distributor → Pharmacy lifecycle
- Operate seamlessly on mobile and desktop (PWA)
- Provide dashboard to display summary of alert metrics with filter options
- Support bilingual operation (English / German) with full RTL

## 2. Problem Statement

Life Science / Pharma customers producing for EU Market are reporting to EU Hub (European Medicine Verification System – EMVS) for track and trace relevant products.
Once products are reported to EU Hub by Manufacturers they get distributed through supply chain to distributors, wholesalers, retailers and at the end to Pharmacy / Hospitals.
 
EU has a point of dispense model, i.e. at Pharmacy / Hospitals, before product is dispensed to patient, it’s scanned to verify and send dispensing event to National system and in case of issue an alert is generated from NMVS system (National Medicine Verification System – Each European country has its own national system which generates an alert based upon defined Alert categories and it gets notified to the manufacturer).
 
Now manufacturers have to handle these alerts on daily basis (for big manufacturers volume of these alerts can be in thousands per day) and they have to manage and respond to these alerts (mostly manually with dedicated FTEs to identify the root cause and provide resolution).
If these alerts are not handled, those medicines can’t be dispensed which have immediate impact

There is a market need for a modern, modular EU FMD alert manager that:

- Serves as one stop shop for displaying the alerts
- user friendly
- Cloud-native
- Extensible to AI and analytics
- With options to trigger uipath automations when a user action is taken.

## 3. Product Vision

Build a next-generation EU FMD alert management platform that:

- has Alerts tab to display all the alerts received with filter options for alert type, month received,date received.
- Has Market configuration tab to add/update/remove Market like Germany, Czech Republic and others
- Dashboard tab to dislay total alerts received, total closed, total open with filters by date range, month, year. 
- Provides option to resolve/close alert by updating comments or accepting comments updated in database by the AI agents in backend. 
- Has option to email NMVO on the root cause explaination for each alert automatically on a button press rather than manually drafting email.
- Has option to filter based on root cause displayed and close the alerts in bulk with option to choose whether to send response to NMVS or not.

## 6. Roles & Personas

The app supports 3 roles across 3 personas. .

### Roles

| Role | Persona | Supervisor |
|------|---------|-----------|
| AlertHandler | Support specialist to handle alerts | No |
| AlertHandler_supervisor | Supervisor who monitors alert resolution | Yes |
| admin | EU FMD management system admin | Yes |


### Supervisor Capabilities

Supervisors have access to all alerthandler functions plus:

- Market Settings page to have each alerthandler view for alerts from respective markets and dashboard display related to those markets alone. 
- Bulk response approval when an alerthandler performs a bulk response for multiple alerts
- Create  new users with alerthandler role alone and assign markets for which the data needs to be displayed for the user. Should not have access to create user with supervisor role. 

### Admin Capabilities

Admin have access to all Supervisor functions plus:

- Backend connection configuration to connect to supabase database. 
- Creating supervisor and Alerthandler users. 
- Configuring Market settings. 

### AlertHandler Operations

- View Alerts displayed in table format with filters
- Update comments against each alerts
- Mark each alert as closed
- Choose to send response to NMVS. 
- Bulk update/mark alerts as closed and send response to NMVS


### Supervisor Operations

- Review all alerts closed with root cause reasons
- Approve bulk alert closure request initiated by AlertHandlers
- Assign market settings to AlertHandlers
- Generate reports based on alert type, market, specific dates, specific months or years, alerts worked by specific AlertHandler.

### Navigation

**Desktop:** Collapsible sidebar with grouped menu sections

### Menu Structure

Menu items are organized into sections and filtered by user role/persona:

1. **Home** — Dashboard with overall metrics of alerts and option to choose market. Alerts should be fetched from backend db and count displayed based on market selected. Filter options to include filters by date, month, year, status, batch. Unique GTIN values should be fetched from Db and displayed as a widget as well.  
2. **Alerts** — Display all alets based on Market, update comments against each alert, mark alert as closed, send response to NMVS

4. **Users** (Supervisor and Admin only) — List all existing user with roles, Create new users, update user details, send password reset link, map role
5. **Admin** (Admin only) — Configure Backdend database connection, create supervisor users and map roles

## 10. Screen Definitions

### Home

- Includes multiple widgets to display the total alerts, total closed, total open and Unique GTIN count with filter options with dropdowns based on Status, Market, Month, Date. Count should be displayed to user based on the role and market assigned to them in market settings. Admin and Supervisor should be able to view count of all markets as well. 
- For AlertHandler, the view should be restricted to what market has been assigned to them without an option to select market. Other filter options should be enabled.
- Include optins to generate reports based on All alerts report, datewise, monthwise, statuswise and when user clicks on it should provide option to save the file in local system folder and let them choose the folder to save and give the same. The output report should be downloaded in csv format. 
- Provide option to export the entire dashboard with all metrics widgets as pdf or share it with email ids. 
- When user clicks on share, it should ask for email to which the export should be sent as pdf and email should be triggered with attachment.  

Rules:
- For coming up with each metrics, it should query the supabase database to query based on status, market etc.,

### Alerts

- Dislay all alerts retrieved from the supabase database as a table display with options to navigate multiple pages and dropdown to select 10,20,50 items per page
- Database would contain columns as below, same columns needs to be displayed for in alerts page.
        - AlertID
        - AlertTimestamp
        - Status (Open, InProgress, Closed, OnHold)
        - ErrorCode
        - TargetMarket (EU, IN, US, UK, BR, ZA, CN, ES)
        - AlertMessage
        - GTIN
        - ExpiryDate
        - SerialNumber
        - BatchName
        - MessageGUID
        - CreatedOn
        - ChangedOn
        - RootCause/Comments
        - AssignedUser (Dropdown with list of all AlertHandlers and Supervisors mapped to respective Market)
- Provide Root cause/Comments column as editable field against each alert record displayed
- Manual input with multi-paste support (comma/newline separated)
- Fetch Alerts button to trigger an API endpoint for an automation in the backend to fetch alert details in ATTP and load it in to supabase db.
